🔄 Luồng thực thi chính (Google Form)

1

Người dùng bấm nút (GUI) hoặc chạy CLI

GUI: App.run(submit) đọc widget → đóng gói params tuple → spawn threading.Thread(_worker). CLI: argparse parse args → dispatch(a).

params = (target, extra, submit, headed, watch)
2

Mở rộng file đầu vào (multi-record expansion)

_worker duyệt danh sách file. File CSV/JSON/TSV (nếu đích ≠ Zalo) → đọc thành list[dict] bằng read_data_file(). N rows → N mục. File ảnh/PDF/Word → giữ nguyên path. Mọi dict hàng được inject __csv_path__.

expanded = [dict_row1, dict_row2, ..., "path/img.jpg", ...]
3

Soi Google Form — resolve_schema()

Mỗi phiên luôn gọi fetch_form() (không đọc cache file). urllib tải HTML form → bóc biến FB_PUBLIC_LOAD_DATA_ → parse JS array → trích title, fields, số trang. Lưu schema file + lưu URL dùng lần sau.

fields = [{id:"entry.123", label:"Họ tên", type:"text", required:True}, ...]
4

Đọc chứng từ — file_to_pages()

Phân nhánh theo extension:

  • .pdf → pdfplumber text (nếu >50 ký tự) hoặc fitz render → ảnh base64
  • .jpg/.png/... → OpenCV process → base64
  • .xlsx/.csv/.docx/.json/... → đọc thẳng thành text/markdown
  • dict (CSV row) → bỏ qua bước này, đi thẳng sang L4b
5

OCR / LLM trích xuất — extract_values()

Kiểm tra _ocr_cache: nếu hit → trả ngay (preview→submit dùng chung). Nếu miss:

  • build_prompt(fields, doc_text) → prompt liệt kê đúng các id/label/type form
  • ChainOCRAdapter.ocr() → Gemini trước (retries=1), nếu fail → OpenAI GPT-4o-mini
  • _strip_json() bóc JSON thuần (loại ``` fence, tìm [{...}])
  • Nếu LLM trả array → đa bản ghi (multi_records); nếu object → 1 bản ghi
  • Kết quả lưu vào _ocr_cache
result = [{"entry.123": "Nguyễn Văn A", "entry.456": "12/03/1984", ...}, ...]
5b

Chuẩn hoá — _normalize_record() [nếu doc là dict từ CSV]

Áp dụng .map.json nếu có. _map_raw_to_fields() ánh xạ tên cột CSV → entry id (exact → NFC+lower → .map.json). normalize_value() chuẩn hoá ngày, khớp option radio/dropdown bằng skeleton matching.

items = [{...field_meta, "value": "Nguyễn Văn A"}, ...] issues = []
6

Điền form — _submit_form_record()

Nếu N > 1 bản ghi: mở 1 Chrome qua browser_context() dùng chung. Với mỗi bản ghi: gọi fill_and_submit_browser() với shared browser → mở tab mới, điền, đóng tab.

  • page.goto(form_url) → wait_for_selector("[role='button']")
  • Duyệt trang: _try_fill_field() → bấm Tiếp (chờ nút cũ hidden + nút mới xuất hiện) → bấm Gửi
  • _confirm_submitted(): wait_for_url(formResponse) — nếu không rời được form → bóc lỗi xác thực
  • Chụp ảnh màn hình vào screenshots/
  • Nếu Playwright thất bại → fallback CUA (Gemini Vision nhìn pixel)
✓

Tổng kết

In N/M bản ghi thành công. Trả mã 0 (tất cả OK) hoặc 1 (có lỗi). GUI mở khoá nút.

Luồng retry + fallback

🔁 Retry trong 1 model (HTTP layer)
  • MAX_API_RETRIES = 3 (standalone), retries=1 khi trong chain
  • 429 quota → break ngay, không retry
  • 429 rate-limit → đọc retryDelay từ response JSON
  • 404/4xx → break ngay
  • 500/503 → retry với backoff 2^n giây
  • Timeout/exception mạng → retry
🔄 Cross-provider fallback (Chain layer)
  • ChainOCRAdapter thử lần lượt từng adapter
  • Adapter đầu (Gemini) fail → in cảnh báo + error detail
  • Chuyển sang adapter sau (OpenAI)
  • GEMINI_MODEL_FALLBACK = [] (không thử model Gemini khác)
  • Kết quả thành công được cache chung
🧩 Prompt engineering
  • Schema-driven: prompt liệt kê đúng các trường form cần
  • Ngày yêu cầu DD/MM/YYYY, option phải khớp danh sách cho sẵn
  • Đa bản ghi: yêu cầu trả [{...}, {...}] nếu nhiều records
  • temperature=0.1 (nhất quán, ít sáng tạo)
  • Text file: nhúng nội dung vào prompt (không gửi ảnh)
  • Image: gửi base64 inline_data jpeg

🎯 Chi tiết các đích điền

🌐 Google Form — Playwright

Cơ chế: Mở Chrome thật (headless hoặc có UI), điền qua DOM selector.

  • Tìm câu hỏi bằng div[role='listitem'] filter has_text=label
  • text/paragraph → textbox.fill()
  • date → input[type="date"].fill() + điền Giờ/Phút nếu có
  • radio/scale → get_by_role("radio", name=val).click()
  • dropdown → click listbox → click option
  • Multi-page: bấm "Tiếp" → chờ nút cũ hidden → chờ nút mới
  • Xác nhận: wait_for_url(formResponse) timeout 12s
  • Thất bại → bóc lỗi xác thực từ DOM → fallback CUA
  • Multi-record: 1 browser, N tab tuần tự

⚡ Cache & Tối ưu hiệu năng

🧠 OCR Session Cache

Tránh gọi LLM 2 lần cho cùng 1 file trong 1 phiên (Xem trước → Điền & Nộp).

  • Key = (doc_path, mtime, fields_hash_8chars)
  • fields_hash: MD5 của list entry_id (nhận biết form đổi)
  • Mtime đổi → cache miss (file bị sửa)
  • Tự động, transparent với caller
  • Chỉ cache khi success
_cache_key()
extract_values()
🌐 Browser Reuse (Multi-record)

Khi nộp N > 1 bản ghi lên Google Form, chỉ mở Chrome 1 lần.

  • browser_context(): context manager giữ browser sống
  • Mỗi bản ghi: mở tab mới, điền, đóng tab
  • Tiết kiệm ~2-3s/bản ghi từ bản ghi thứ 2
  • Chỉ áp dụng khi N > 1 và không dùng CUA
  • Browser tự đóng khi context manager thoát
browser_context()
run_form()
🔀 Maker-Checker Pattern

2 bước: Xem trước (không gửi) → Điền & Nộp (gửi thật).

  • Xem trước: submit=False → OCR + hiện giá trị, KHÔNG mở browser form
  • Điền & Nộp: submit=True → dùng OCR cache → mở browser → điền
  • Xác nhận dialog trước khi nộp thật
  • LLM chỉ gọi 1 lần nhờ _ocr_cache
📄 PDF Text-First Strategy

Tránh tốn API vision token cho PDF đã có text layer.

  • pdfplumber bóc text từng trang trước
  • >50 ký tự → gửi text (rẻ, nhanh, chính xác hơn)
  • ≤50 ký tự → PDF scan → fitz render 2x → gửi ảnh
  • Bảng trong PDF → chuyển thành markdown table
⏱️ Playwright Navigation Optimization

Thay chờ cứng 900ms bằng chờ sự kiện thực.

  • Sau bấm Tiếp: next_b.wait_for(state="hidden", 3s)
  • Rồi: wait_for_selector("[role='button']", 5s)
  • Chờ đúng animation xong, không chờ dư
  • Tiết kiệm 0.5-0.7s/trang trung bình
